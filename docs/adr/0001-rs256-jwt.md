# ADR-0001: Choose RS256 over HS256 for JWT signing

**Status**: Accepted
**Date**: 2026-07-22
**Deciders**: Platform team

## Context

Agent System v0.1.0 ships JWT auth with HS256 (symmetric secret).
The secret is shared between every signer and every verifier —
issuer, replicas, and external services that need to verify
tokens all need to hold the same `AUTH_SECRET` (or one of the
rotation keys in `AUTH_SECRETS`).

This works for a single-issuer / single-replica deployment. It
breaks down in two scenarios we want to support:

1. **Multi-tenant SaaS with multiple issuer services** (e.g. one
   per region). Each issuer would need to hold the verification
   secret too, multiplying the attack surface.
2. **External verifiers** — partner integrations, microservices,
   Jupyter notebooks for data scientists — need to verify our
   tokens without us shipping a secret to them.

The HS256 multi-key rotation (`AUTH_SECRETS`) helps the rotation
story but doesn't solve the trust-boundary problem: every verifier
is a potential exfiltrator of the symmetric secret.

## Decision

We support both algorithms and **recommend RS256 for any
deployment beyond a single replica**:

- `AuthService` auto-detects the algorithm from env at construction:
  - `AUTH_PRIVATE_KEY` set → RS256 (asymmetric, recommended).
  - Otherwise → HS256 (legacy, backward-compatible).
- Public keys are distributed via `GET /api/auth/jwks` (RFC 7517).
  External verifiers fetch the JWKS once, cache, and verify locally.
- The signing key (`AUTH_PRIVATE_KEY`) only ever lives on issuing
  replicas — verifier-only replicas omit it and rely on
  `AUTH_PUBLIC_KEYS`.
- `scripts/gen_rsa_keys.py` generates 2048/3072/4096-bit RSA
  keypairs with a JWKS preview, ready for `.env` deployment.

RS256 introduces no runtime overhead vs HS256 in practice (modern
CPUs handle 2048-bit RSA verify in ~100µs, far below request
latency budgets).

## Alternatives considered

### Stay on HS256 with manual key distribution to verifiers
Doesn't scale: any verifiers we ship a secret to become a
compromise vector. Not recommended for SaaS, partners, or any
multi-tenant scenario.

### Move to a third-party identity provider (Auth0, Cognito, Keycloak)
Offloads the auth problem entirely, but:
- Adds a hard external dependency (auth outages = service outage).
- Costs real money at scale.
- The current `User` / `TenantContext` / `RBAC` model is custom
  enough that integration work is non-trivial.

Worth considering in the future if/when SAML / OIDC support
becomes a real requirement; not now.

### EdDSA (Ed25519)
Faster than RS256, smaller signatures, but:
- Requires modern crypto libraries everywhere (verifiers may be
  on old runtimes).
- Far less toolchain support — JWT libraries all support RS256;
  EdDSA support is patchy.

## Consequences

### Positive
- External verifiers can verify tokens without sharing any secret.
- Multi-tenant / multi-region deployments become straightforward:
  sign in one place, verify everywhere.
- Key rotation is asymmetric — keep old public keys in
  `AUTH_PUBLIC_KEYS`, retire the old private key after all
  tokens using it have expired.
- Backward compatible — existing HS256 deployments keep working
  without any change.

### Negative
- More env vars to manage (`AUTH_PRIVATE_KEY`,
  `AUTH_PUBLIC_KEYS`, `AUTH_SIGNING_KID`).
- Operators must run `scripts/gen_rsa_keys.py` (or equivalent) on
  first deploy — small UX cost vs HS256's "type a random string".
- Private key is now a load-bearing secret that must be stored in
  a secret manager (Vault, AWS Secrets Manager) rather than a plain
  `.env` file. Mitigation: `PRODUCTION.md` §"JWT signing" documents
  this requirement.
- Adds ~80 lines of code to `core/auth/jwt.py`. Acceptable.
